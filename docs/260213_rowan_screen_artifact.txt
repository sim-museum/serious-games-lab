Root Cause Analysis

The black box you're seeing is almost certainly a DirectDraw primary surface or overlay that MiG Alley creates for its 2D UI layer. Looking at the code structure:

The Technical Issue

    Dual Rendering Architecture: MiG Alley (released 1999) uses a mixed rendering approach:
        DirectDraw for 2D UI elements (menus, HUD, OCX controls)
        Direct3D for the 3D flight simulation view

    OCX Controls: The game loads 13 custom Rowan Software ActiveX controls (RTabs.ocx, RListBox.ocx, Rbutton.ocx, etc.) that create Windows HWND child windows for UI elements. Under Wine,
    these may create unrendered or improperly composited overlay windows.

    DirectDraw Surface Issue: Games from this era typically created a DirectDraw "primary surface" at a fixed resolution (often 640x480, 800x600, or 1024x768). Wine's ddraw.dll must
    translate this to modern OpenGL/Vulkan, and the black box is likely:
        A DirectDraw clipper region that isn't being properly hidden
        A secondary surface meant to hold UI that Wine is rendering as opaque black
        The "software driver" fallback surface the game creates (mentioned in readme.txt line 157)

    Why Dual Monitors Help

    When you have two monitors of equal resolution, Wine's DirectDraw wrapper positions the black overlay surface on the second monitor because:
        DirectDraw exclusive mode captures one display
        The overlay surface coordinates extend beyond the primary display bounds
        Wine interprets the surface position literally, pushing it to monitor 2

    Why High Resolution Helps

    The game's maximum supported resolution is 1152x864 (per migAlley.sh line 71-72). On a 4K display:
        The black box remains at its original size (e.g., 1024x768)
        It occupies only a small portion of the screen
        The 3D view renders at higher resolution, making the overlap proportionally smaller

● Here's what each registry setting controls:

Direct3D Settings

(HKEY_CURRENT_USER\Software\Wine\Direct3D)

DirectDrawRenderer = "opengl"

    Forces Wine to use OpenGL (instead of GDI) for translating DirectDraw calls

    DirectDraw was designed for direct framebuffer access on Windows; Wine must translate this to something modern Linux understands

    OpenGL mode uses hardware acceleration and proper surface compositing

    Alternative value "gdi" uses software rendering, which is slower but sometimes more compatible

    OffscreenRenderingMode = "fbo"

    FBO = Framebuffer Objects (an OpenGL feature)

    Controls how Wine renders surfaces that aren't displayed directly on screen

    With "fbo", off-screen surfaces (like the UI overlay buffer) render to GPU memory and composite properly

    Alternative "backbuffer" mode can cause the exact black box artifact you're seeing because it allocates a separate backbuffer that may not composite correctly with the primary 3D surface

    DirectDraw Settings

    (HKEY_CURRENT_USER\Software\Wine\DirectDraw)

    DefaultSurfaceType = "gdi"

    Controls how DirectDraw primary/overlay surfaces are created

    "gdi" mode makes DirectDraw surfaces behave like regular Windows GDI windows

    This means the overlay surface gets proper window manager integration rather than trying to do exclusive fullscreen tricks

    The black box likely appears because Wine's default DirectDraw surface type creates an opaque, unmanaged surface that sits on top of everything

    Why These Might Help

    MiG Alley creates at least two surfaces:
        A Direct3D surface for the 3D cockpit/world view
        A DirectDraw surface for 2D UI elements (HUD, menus, OCX controls)

    The black box is likely surface #2 being rendered as an opaque black rectangle because Wine doesn't know what to fill it with, or it's being positioned/sized incorrectly. The registry
    tweaks change how Wine allocates and composites these surfaces together.

The Root Cause

MiG Alley likely does something like this:

    Creates a DirectDraw primary surface in exclusive fullscreen mode

    Creates a DirectDraw clipper attached to the main window

    Creates child windows (OCX controls) for 2D UI elements

    The clipper is supposed to define which regions of the primary surface are visible vs. obscured by child windows

    Wine 9.0's problem: The clipper region or the child window z-ordering isn't being handled correctly, so:
        The "clipped" region renders as black instead of being transparent/hidden
        In virtual desktop mode, child window coordinates are still calculated relative to absolute screen position (0,0) rather than the virtual desktop origin, so they appear off-screen or
        clipped

    Why Older Wine Works Better

    Wine 4.21 (circa 2019) used a different DirectDraw implementation that:
        Handled exclusive fullscreen mode more like Windows 98/XP did
        Had different (more permissive) child window coordinate handling
        Used an older wined3d backend with different surface compositing logic

    Between Wine 4.x and 9.x, the DirectDraw code was significantly refactored for "correctness" which broke compatibility with games that relied on quirky legacy behavior.
